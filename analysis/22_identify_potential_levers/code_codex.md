# Code Review (codex)

## Bugs Found

- **B1 — The review contract is internally contradictory, so the schema itself trains models to emit the wrong shape.** In `/Users/neoneye/git/PlanExeGroup/PlanExe/worker_plan/worker_plan_internal/lever/identify_potential_levers.py:92-98`, `review_lever` is described as "two sentences" but the only example is a single sentence beginning `This lever governs...`. The system prompt repeats the same contradiction at `/Users/neoneye/git/PlanExeGroup/PlanExe/worker_plan/worker_plan_internal/lever/identify_potential_levers.py:214-218`: it says "two sentences" and then gives a one-sentence example with the same opening. Because these field descriptions are part of the structured-output schema, this is not just stylistic drift — the prompt assembly itself anchors the one-sentence `This lever ...` pattern seen in the insight files.

- **B2 — Partial-call failures are swallowed inside `IdentifyPotentialLevers.execute()`, and the runner records them as normal success.** In `/Users/neoneye/git/PlanExeGroup/PlanExe/worker_plan/worker_plan_internal/lever/identify_potential_levers.py:299-318`, any exception after at least one successful call is logged and then skipped with `continue`, but the returned object at `/Users/neoneye/git/PlanExeGroup/PlanExe/worker_plan/worker_plan_internal/lever/identify_potential_levers.py:352-359` carries no failed-call index, error class, or error ID. Then `/Users/neoneye/git/PlanExeGroup/PlanExe/self_improve/runner.py:126-139` returns `status="ok"` whenever some output was saved, and `/Users/neoneye/git/PlanExeGroup/PlanExe/self_improve/runner.py:322-337` emits `run_single_plan_complete` plus a coarse `partial_recovery` count. That means `events.jsonl` cannot tell whether the missing third call was a timeout, provider failure, JSON parse failure, or schema rejection.

- **B3 — The option-quality contract is never validated, so label-length options pass unchanged.** The schema text says options must be complete strategic sentences in `/Users/neoneye/git/PlanExeGroup/PlanExe/worker_plan/worker_plan_internal/lever/identify_potential_levers.py:88-90`, and the follow-up prompt repeats that requirement at `/Users/neoneye/git/PlanExeGroup/PlanExe/worker_plan/worker_plan_internal/lever/identify_potential_levers.py:271-274`, but the only implemented validator at `/Users/neoneye/git/PlanExeGroup/PlanExe/worker_plan/worker_plan_internal/lever/identify_potential_levers.py:114-125` checks list length only. There is no structural gate for sentence-ness, minimum content, or even a crude length floor, so outputs like short campaign labels are accepted as fully valid levers.

- **B4 — `OPTIMIZE_INSTRUCTIONS` says every lever needs a conservative option, but the active prompt never actually asks for one.** The optimization notes explicitly require "at least one conservative, low-risk path" at `/Users/neoneye/git/PlanExeGroup/PlanExe/worker_plan/worker_plan_internal/lever/identify_potential_levers.py:48-52`. But the active system prompt at `/Users/neoneye/git/PlanExeGroup/PlanExe/worker_plan/worker_plan_internal/lever/identify_potential_levers.py:202-208` asks for one "unconventional or non-obvious approach" and says nothing about a conservative baseline, and the incremental prompt for calls 2/3 at `/Users/neoneye/git/PlanExeGroup/PlanExe/worker_plan/worker_plan_internal/lever/identify_potential_levers.py:269-275` also omits that requirement. This is a direct code-level mismatch between the documented quality goal and the actual prompt the model receives.

## Suspect Patterns

- **S1 — Cross-call anti-repetition is name-based only, not concept-based.** Follow-up calls blacklist only prior lever names at `/Users/neoneye/git/PlanExeGroup/PlanExe/worker_plan/worker_plan_internal/lever/identify_potential_levers.py:269-275`, and post-merge cleanup skips only exact string matches at `/Users/neoneye/git/PlanExeGroup/PlanExe/worker_plan/worker_plan_internal/lever/identify_potential_levers.py:329-336`. That design prevents literal duplicates, but it does nothing against "Standards" vs. "Standards Enforcement" vs. "Standards Harmonization"-style renames. This may be acceptable in the full pipeline because semantic dedup happens later, but it is a poor fit for the prompt-lab artifact being analyzed here, which stops at `002-10-potential_levers.json`.

- **S2 — The runner fingerprints only the external system prompt, not the effective prompt logic that actually changed.** `/Users/neoneye/git/PlanExeGroup/PlanExe/self_improve/runner.py:373-389` records `system_prompt_sha256`, but follow-up-call wording is assembled dynamically inside `/Users/neoneye/git/PlanExeGroup/PlanExe/worker_plan/worker_plan_internal/lever/identify_potential_levers.py:269-275`. Any code change to those inline instructions changes model behavior without changing `meta.json`, which makes prompt-lab experiments harder to compare and weakens post-hoc attribution.

- **S3 — Review validation enforces only placeholder absence and minimum length, not the shape the prompt promises.** `/Users/neoneye/git/PlanExeGroup/PlanExe/worker_plan/worker_plan_internal/lever/identify_potential_levers.py:127-142` deliberately removed fragile English keyword checks, which is good, but it also dropped any enforcement of the claimed two-sentence structure. That leaves the code in an awkward middle ground: the prompt promises a format, but the validator knowingly accepts something else.

## Trace to Insight Findings

- **B1 + S3 explain the review-shape findings.** The insight files report poor two-sentence compliance and extreme repetition of `This lever ...` openings. That maps directly to the contradictory one-sentence example and the validator that never checks sentence count.

- **B2 explains the run-60 partial-recovery blind spot.** The insight files note plans with only 2 of 3 calls succeeding and incomplete `events.jsonl` telemetry. That is exactly what the `continue` path in `IdentifyPotentialLevers.execute()` plus the runner's coarse `status="ok"` reporting produces.

- **B3 explains the terse, label-like option failures.** The insight files' short options are not surprising given that the code enforces `len(options) == 3` and nothing else.

- **B4 explains the missing conservative path in weaker-model outputs.** The insight files call out lever sets that offer only variants of the same risky approach. The source says conservative options matter, but the active prompt never actually requests them.

- **S1 explains the semantic-duplication clusters.** The insight files' repeated lever families are exactly the kind of failure expected when the prompt bans only reused names and the merge step removes only exact-name duplicates.

- **I5 targets the unsupported percentages / budget-split regressions.** The insight files show that stronger prompt wording helps some models but not all. A post-parse grounding audit would catch the remaining pseudo-precise claims that are structurally valid JSON.

- **S2 + I7 explain why code-only prompt changes are hard to audit in prompt-lab history.** The runner stores the external system prompt hash, but not the inline call-2/3 prompt logic that materially changes behavior.

## Summary

- The strongest confirmed code issues are: contradictory review instructions, swallowed partial-call failures, missing validation for label-like options, and a real mismatch between `OPTIMIZE_INSTRUCTIONS` and the prompt actually sent to the model.
- The clearest quality root causes from the insight files are therefore mostly **prompt/spec consistency bugs** and **observability gaps**, not JSON-schema failures.
- The highest-leverage follow-ups are: fix the review contract, persist per-call failure metadata, add structural option-quality checks, and move anti-duplication plus conservative-option requirements into the actual prompt path used at runtime.
