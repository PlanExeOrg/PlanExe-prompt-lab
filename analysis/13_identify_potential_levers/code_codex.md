## Bugs Found

- **B1 — PR #289 hardcodes English review markers, so valid non-English outputs will be rejected.** On the PR branch, `worker_plan/worker_plan_internal/lever/identify_potential_levers.py:86` adds `check_review_format()`, and `worker_plan/worker_plan_internal/lever/identify_potential_levers.py:95` / `worker_plan/worker_plan_internal/lever/identify_potential_levers.py:97` reject any `review_lever` string that does not literally contain `Controls ` and `Weakness:`. That is not a schema check; it is an English-literal check. In this repo, prompt-optimizer work explicitly says validators must be language-agnostic because plans may be multilingual: `prompt_optimizer/AGENTS.md:165`, `prompt_optimizer/AGENTS.md:166`, `prompt_optimizer/AGENTS.md:168`. The PR therefore fixes the English experiment set while introducing a correctness bug for non-English plans.

- **B2 — The PR validates levers inside a whole-call `DocumentDetails` parse, so one bad lever discards the entire 5–7 lever batch.** On the PR branch, `worker_plan/worker_plan_internal/lever/identify_potential_levers.py:101` defines `DocumentDetails.levers` as `list[Lever]`, and `worker_plan/worker_plan_internal/lever/identify_potential_levers.py:249` / `worker_plan/worker_plan_internal/lever/identify_potential_levers.py:250` ask the structured LLM to parse the whole response as one `DocumentDetails`. Once `check_option_count()` or `check_review_format()` raises for a single lever, the entire call fails before any of the sibling levers are available to the caller. This is the direct code-level cause of the call-level rejection cascade reported in `insight_claude.md` for llama3.1 and haiku.

- **B3 — `execute()` throws away earlier successful calls when a later call fails, so validator errors become full-plan failures.** `worker_plan/worker_plan_internal/lever/identify_potential_levers.py:194` hardcodes three calls. Successful calls append to `responses` at `worker_plan/worker_plan_internal/lever/identify_potential_levers.py:243`, but any later exception is wrapped and re-raised at `worker_plan/worker_plan_internal/lever/identify_potential_levers.py:236` / `worker_plan/worker_plan_internal/lever/identify_potential_levers.py:240`. Flattening into the final lever list happens only after the loop at `worker_plan/worker_plan_internal/lever/identify_potential_levers.py:246` / `worker_plan/worker_plan_internal/lever/identify_potential_levers.py:274`. So if call 3 fails after calls 1–2 succeeded, the method returns nothing and the runner writes no raw/clean lever artifact. This is the second layer of the cascade bug described in the insight files.

- **B4 — The PR’s “review format” validator does not actually validate the claimed format; it only validates marker presence.** On the PR branch, `worker_plan/worker_plan_internal/lever/identify_potential_levers.py:88` says it is rejecting missing required format, but the implementation only checks two substrings at `worker_plan/worker_plan_internal/lever/identify_potential_levers.py:95` and `worker_plan/worker_plan_internal/lever/identify_potential_levers.py:97`. It does not verify two-sentence shape, `vs.` punctuation, the `Controls [A] vs. [B]. Weakness: ...` order, or placeholder leakage. The declared contract is stricter in `worker_plan/worker_plan_internal/lever/identify_potential_levers.py:53`, `worker_plan/worker_plan_internal/lever/identify_potential_levers.py:54`, `worker_plan/worker_plan_internal/lever/identify_potential_levers.py:55`, and the active prompt used in this batch repeats that stricter shape at `prompts/identify_potential_levers/prompt_2_75f59ab464162a827807a68b029321534ae1686f6fc340e278c7881d3b334d39.txt:25` / `prompts/identify_potential_levers/prompt_2_75f59ab464162a827807a68b029321534ae1686f6fc340e278c7881d3b334d39.txt:26`. This explains why the targeted missing-marker defects disappear after the PR while 71 “non-exact review format” cases still pass.

- **B5 — The runner keeps almost none of the useful failure artifacts, which is why validator-triggered failures look like empty output dirs.** `prompt_optimizer/runner.py:118` / `prompt_optimizer/runner.py:121` write `002-9-potential_levers_raw.json` and `002-10-potential_levers.json` only on full success. On failure, `prompt_optimizer/runner.py:130` / `prompt_optimizer/runner.py:137` keep only `str(e)` in `PlanResult.error`, and `prompt_optimizer/runner.py:139` / `prompt_optimizer/runner.py:143` clear usage-metric state and delete `track_activity.jsonl`. Combined with B3, this means a plan can spend tens of seconds generating useful partial content and still leave only `activity_overview.json` and `usage_metrics.jsonl`, exactly matching the empty-output evidence in `insight_claude.md`.

## Suspect Patterns

- **S1 — The active registered prompt likely teaches the exact label leakage later counted as an output defect.** The runner loads an external prompt file at `prompt_optimizer/runner.py:436`; for this analysis batch the selected file is recorded at `analysis/13_identify_potential_levers/meta.json:2`. That prompt tells the model to show progression `conservative → moderate → radical` at `prompts/identify_potential_levers/prompt_2_75f59ab464162a827807a68b029321534ae1686f6fc340e278c7881d3b334d39.txt:15`, then separately forbids `Conservative:`, `Moderate:`, and `Radical:` labels at `prompts/identify_potential_levers/prompt_2_75f59ab464162a827807a68b029321534ae1686f6fc340e278c7881d3b334d39.txt:32`. That contradiction is a plausible source of the label leakage observed in the after runs.

- **S2 — The prompt and field descriptions expose literal bracket placeholders, and nothing rejects leaked placeholders downstream.** The review schema description uses `[Tension A]`, `[Tension B]`, and `[specific factor]` at `worker_plan/worker_plan_internal/lever/identify_potential_levers.py:54` / `worker_plan/worker_plan_internal/lever/identify_potential_levers.py:55`, and the active prompt repeats the same bracketed scaffold at `prompts/identify_potential_levers/prompt_2_75f59ab464162a827807a68b029321534ae1686f6fc340e278c7881d3b334d39.txt:25` / `prompts/identify_potential_levers/prompt_2_75f59ab464162a827807a68b029321534ae1686f6fc340e278c7881d3b334d39.txt:26`. Because `check_review_format()` only looks for `Controls ` and `Weakness:`, bracket placeholders can leak straight through.

- **S3 — Cross-call diversity is enforced only on lever names, not on the underlying strategic content.** Follow-up calls only receive a denylist of previously generated names at `worker_plan/worker_plan_internal/lever/identify_potential_levers.py:203` / `worker_plan/worker_plan_internal/lever/identify_potential_levers.py:206`. The model never sees the already-covered option content, trade-offs, or consequences. That makes it easy to avoid literal name reuse while still repeating the same “radical” business idea across multiple levers, which fits the repeated blockchain trope seen in sovereign-identity outputs.

- **S4 — The schema asks every call for extra wrapper prose that the final artifact throws away, increasing response size without downstream value.** `DocumentDetails.summary` is required at `worker_plan/worker_plan_internal/lever/identify_potential_levers.py:85`, every one of the three calls requests a full `DocumentDetails` object at `worker_plan/worker_plan_internal/lever/identify_potential_levers.py:222`, but `save_clean()` writes only the cleaned lever list at `worker_plan/worker_plan_internal/lever/identify_potential_levers.py:306` / `worker_plan/worker_plan_internal/lever/identify_potential_levers.py:308`. That extra token budget is a plausible contributor to truncation and self-editing artifacts on weaker models, though I would verify with prompt-length telemetry before treating it as a confirmed bug.

## Improvement Opportunities

- **I1 — Replace marker-literal validation with language-agnostic structure.** Instead of checking for English words in one free-text field, make the review machine-readable or at least parseable as separate fields such as `tension_a`, `tension_b`, and `weakness`. If the single-string format must stay, add a light post-save linter rather than a hard English-only validator.

- **I2 — Recover partial results at both failure boundaries.** First, if a later call fails after at least one successful call, keep the earlier `responses` instead of re-raising immediately from `worker_plan/worker_plan_internal/lever/identify_potential_levers.py:236`. Second, stop validating the whole `DocumentDetails` as an all-or-nothing batch; validate or salvage levers individually so one malformed lever does not zero out six good siblings.

- **I3 — Add a `consequences` contamination repair/validator.** The field description already says review text does not belong in `consequences` at `worker_plan/worker_plan_internal/lever/identify_potential_levers.py:42`, but nothing enforces that before the value is copied into the saved artifact at `worker_plan/worker_plan_internal/lever/identify_potential_levers.py:264`. A simple split-or-reject pass would directly target the contamination that persists unchanged before and after the PR.

- **I4 — Add stricter linting for template leakage that the PR currently misses.** If the current validator stays, add a second pass that flags or repairs bracket placeholders, `Conservative:` / `Moderate:` / `Radical:` prefixes, and review strings that use `vs` instead of `vs.`. Those are the surviving format defects the PR left untouched.

- **I5 — Preserve failure forensics in the runner.** On exceptions, keep `track_activity.jsonl`, save whatever raw model text or partial parse state is available, and write a small error-sidecar file instead of only returning `str(e)` from `prompt_optimizer/runner.py:130`. That will make future quality regressions much easier to diagnose.

- **I6 — Soften the “radical option must include emerging tech/business model” instruction.** The current prompt makes that a hard requirement at `prompts/identify_potential_levers/prompt_2_75f59ab464162a827807a68b029321534ae1686f6fc340e278c7881d3b334d39.txt:39`, which is a plausible driver of repeated blockchain answers. Requiring novelty or non-obviousness is safer than requiring an “emerging tech/business model” every time.

## PR Review

- **Intent match:** The PR does fix the two narrow silent-schema gaps it set out to fix. `check_option_count()` on the PR branch at `worker_plan/worker_plan_internal/lever/identify_potential_levers.py:75` / `worker_plan/worker_plan_internal/lever/identify_potential_levers.py:82` closes the shipped-2-options bug, and `check_review_format()` at `worker_plan/worker_plan_internal/lever/identify_potential_levers.py:88` / `worker_plan/worker_plan_internal/lever/identify_potential_levers.py:95` closes the shipped-missing-marker bug.

- **Main gap:** The PR does not address the already-existing lossiness in `IdentifyPotentialLevers.execute()`. Because validation happens inside whole-call structured parsing and the loop re-raises on any later failure, the new validators convert malformed subparts into whole-plan dropouts. That behavior is not an accidental side effect of the experiments; it follows directly from the control flow in `worker_plan/worker_plan_internal/lever/identify_potential_levers.py:222`, `worker_plan/worker_plan_internal/lever/identify_potential_levers.py:236`, and `worker_plan/worker_plan_internal/lever/identify_potential_levers.py:246`.

- **Implementation bug in the PR itself:** The new review validator is not language-agnostic and does not fully implement the prompt contract. It is therefore both too strict in the wrong dimension (English-only markers) and too weak in the dimension the insight files still show as failing (exact shape, punctuation, placeholders).

- **Assessment:** Keep the idea, but not the implementation as-is. I would treat PR #289 as **conditional**: pair it with partial-result recovery, and replace the English keyword check with a language-agnostic structure or a non-fatal linter before merging.

## Summary

PR #289 fixes the exact defects it targeted, but the code review shows three important gaps.

First, the new `review_lever` validator is English-literal, which is a real correctness bug for a multilingual system. Second, the validator is inserted into an all-or-nothing `DocumentDetails` parse, and `execute()` still discards earlier successful calls when a later call fails; together those two behaviors explain the new cascade failures. Third, the validator only checks marker presence, so it leaves most of the remaining quality issues untouched: exact-format drift, bracket placeholders, consequence contamination, and template leakage.

If the next change set only does one thing, it should be **partial-result recovery plus per-lever salvage**. That addresses the new hard failures without backing out the data-integrity benefit of the PR.
