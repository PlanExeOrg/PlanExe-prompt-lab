# Code Review (codex)

## Bugs Found

- **B1 — The live system prompt contains the exact metric phrase that later leaks into outputs.** The hard-coded instruction includes the worked example `"Systemic: 25% faster scaling through..."`, so weaker models are being shown the exact string they later copy. That is a prompt-construction bug in the shipped step, not just a model quirk. Refs: `/Users/neoneye/git/PlanExeGroup/PlanExe/worker_plan/worker_plan_internal/lever/identify_potential_levers.py:93-96`.

- **B2 — The step does not encode or enforce its own output contract, and one part of the schema contradicts the prompt.** The prompt says each response must contain exactly 5 levers and each lever exactly 3 options, but the schema uses unconstrained `list[Lever]` and `list[str]`, while both `Lever.options` and `LeverCleaned.options` still describe `2-5 options`. The only validator on `options` parses stringified JSON; it never checks count, prefixes, placeholders, or consequence/review format. After that, all three batches are blindly flattened and written to `002-10-potential_levers.json`, and the runner marks the plan `ok` with no post-merge validation. This directly explains the 16-item/20-item files, split 2-option + 1-option levers, missing chain labels, and prefix/placeholder leakage surviving as “successful” runs. Refs: `/Users/neoneye/git/PlanExeGroup/PlanExe/worker_plan/worker_plan_internal/lever/identify_potential_levers.py:33-51`, `/Users/neoneye/git/PlanExeGroup/PlanExe/worker_plan/worker_plan_internal/lever/identify_potential_levers.py:57-59`, `/Users/neoneye/git/PlanExeGroup/PlanExe/worker_plan/worker_plan_internal/lever/identify_potential_levers.py:78-80`, `/Users/neoneye/git/PlanExeGroup/PlanExe/worker_plan/worker_plan_internal/lever/identify_potential_levers.py:203-219`, `/Users/neoneye/git/PlanExeGroup/PlanExe/worker_plan/worker_plan_internal/lever/identify_potential_levers.py:257-259`, `/Users/neoneye/git/PlanExeGroup/PlanExe/prompt_optimizer/runner.py:118-129`.

- **B3 — Cross-call generation is under-specified: follow-up turns only say `"more"`, so the code never tells the model what has already been covered.** `IdentifyPotentialLevers.execute()` already has all earlier batches in memory, but the continuation loop never feeds back covered lever names/themes, never says “do not repeat”, and never checks uniqueness before merge. The result is that call 2 and call 3 are effectively “continue brainstorming” turns, not controlled requests for five new strategic dimensions. That is a concrete logic error behind the repeated governance/resource/information themes and duplicate lever names across batches. Refs: `/Users/neoneye/git/PlanExeGroup/PlanExe/worker_plan/worker_plan_internal/lever/identify_potential_levers.py:155-159`, `/Users/neoneye/git/PlanExeGroup/PlanExe/worker_plan/worker_plan_internal/lever/identify_potential_levers.py:163-200`, `/Users/neoneye/git/PlanExeGroup/PlanExe/worker_plan/worker_plan_internal/lever/identify_potential_levers.py:203-219`.

- **B4 — The runner never validates model aliases before launching the batch, so one bad model name is rediscovered once per plan.** `run_single_plan()` constructs `LLMModelFromName` objects from raw strings inside each plan execution, and `_resolve_workers()` silently skips unknown names instead of treating them as fatal. As a result, a missing model alias turns into five plan failures rather than one immediate run-level error. That matches the all-plans-lost configuration failure in run 09. Refs: `/Users/neoneye/git/PlanExeGroup/PlanExe/prompt_optimizer/runner.py:93-94`, `/Users/neoneye/git/PlanExeGroup/PlanExe/prompt_optimizer/runner.py:224-254`, `/Users/neoneye/git/PlanExeGroup/PlanExe/prompt_optimizer/runner.py:315-389`.

## Suspect Patterns

- **S1 — Assistant history is fed back as a Python `dict`, not an intentionally serialized message.** The continuation loop appends `result["chat_response"].raw.model_dump()` directly as assistant `content`. If a backend expects string chat content, this may be coerced inconsistently across providers and could help explain odd continuation behavior on weaker models. Refs: `/Users/neoneye/git/PlanExeGroup/PlanExe/worker_plan/worker_plan_internal/lever/identify_potential_levers.py:193-197`.

- **S2 — `_resolve_workers()` does not use the same config-selection path as actual LLM creation.** It merges every `llm_config/*.json` file it can find, but it has no notion of the active profile or runtime selection logic. That means worker-count planning and model-availability assumptions can drift from what `get_llm()` will actually allow at execution time. Refs: `/Users/neoneye/git/PlanExeGroup/PlanExe/prompt_optimizer/runner.py:224-254`.

- **S3 — `run_single_plan()` collapses all failures into a plain string and stops there.** The runner records only `status="error"` plus `str(e)`, so it does not distinguish invalid-model failures, empty-JSON extraction failures, truncation, or contract-broken-but-parseable outputs. That makes targeted retries and downstream diagnosis harder than necessary. Refs: `/Users/neoneye/git/PlanExeGroup/PlanExe/prompt_optimizer/runner.py:130-137`, `/Users/neoneye/git/PlanExeGroup/PlanExe/prompt_optimizer/runner.py:298-311`.

## Improvement Opportunities

- **I1 — Remove the literal metric exemplar from the prompt.** Replace `"Systemic: 25% faster scaling through..."` with a structural cue such as `"Systemic: [specific quantified impact] through [mechanism]"`. That keeps the guidance while removing the exact phrase that run 10 copied verbatim. Refs: `/Users/neoneye/git/PlanExeGroup/PlanExe/worker_plan/worker_plan_internal/lever/identify_potential_levers.py:93-96`.

- **I2 — Turn the prompt contract into executable validation.** Add `min_length=max_length=3` for `options`, `min_length=max_length=5` for `levers`, and step-specific validators for consequence labels (`Immediate:` / `Systemic:` / `Strategic:`), `review_lever` shape, forbidden prefixes, and bracket placeholders. Then add a post-merge check that rejects totals other than 15 before the runner records success. Refs: `/Users/neoneye/git/PlanExeGroup/PlanExe/worker_plan/worker_plan_internal/lever/identify_potential_levers.py:33-51`, `/Users/neoneye/git/PlanExeGroup/PlanExe/worker_plan/worker_plan_internal/lever/identify_potential_levers.py:57-59`, `/Users/neoneye/git/PlanExeGroup/PlanExe/worker_plan/worker_plan_internal/lever/identify_potential_levers.py:203-219`, `/Users/neoneye/git/PlanExeGroup/PlanExe/prompt_optimizer/runner.py:118-129`.

- **I3 — Make follow-up turns stateful instead of sending bare `"more"`.** Pass the names/themes already generated and say they are closed topics, e.g. `"Generate 5 additional levers. Already covered: ... Do not repeat these themes or names."` This is the highest-leverage change for the cross-call duplication seen across almost every successful model. Refs: `/Users/neoneye/git/PlanExeGroup/PlanExe/worker_plan/worker_plan_internal/lever/identify_potential_levers.py:155-159`, `/Users/neoneye/git/PlanExeGroup/PlanExe/worker_plan/worker_plan_internal/lever/identify_potential_levers.py:163-200`.

- **I4 — Fail fast on invalid model names before processing any plans.** Resolve all requested models once at run start, using the same availability logic as actual LLM creation, and abort the run immediately if any alias is unavailable. That converts run-09-style failures from five wasted plan attempts into one actionable startup error. Refs: `/Users/neoneye/git/PlanExeGroup/PlanExe/prompt_optimizer/runner.py:93-94`, `/Users/neoneye/git/PlanExeGroup/PlanExe/prompt_optimizer/runner.py:224-254`, `/Users/neoneye/git/PlanExeGroup/PlanExe/prompt_optimizer/runner.py:315-389`.

- **I5 — Add a step-level retry/repair path for extraction and truncation failures.** Right now the runner makes one attempt at `IdentifyPotentialLevers.execute()` and turns any exception into a terminal plan error. A bounded retry for empty/truncated structured output, or a repair path for wrapper objects whose `levers` field is intact, would improve runs like nemotron and the truncated gpt-oss sample. Refs: `/Users/neoneye/git/PlanExeGroup/PlanExe/prompt_optimizer/runner.py:111-137`.

- **I6 — Serialize continuation context intentionally.** Feed back either a compact JSON string or a curated summary of prior lever names/themes rather than a raw `model_dump()` object. This would make multi-call behavior less provider-dependent and easier to reason about. Refs: `/Users/neoneye/git/PlanExeGroup/PlanExe/worker_plan/worker_plan_internal/lever/identify_potential_levers.py:193-197`.

- **I7 — Add explicit brevity targets or enforce length bounds.** The current prompt is strict about count but loose about field length, so Claude can remain fully compliant while overshooting the likely target size by 2–3×. Length guidance in the prompt, plus optional validators on field size, would reduce token cost and truncation risk without sacrificing structure. Refs: `/Users/neoneye/git/PlanExeGroup/PlanExe/worker_plan/worker_plan_internal/lever/identify_potential_levers.py:88-126`.

## Trace to Insight Findings

- **Run 10 template leakage (`"25% faster scaling through"` repeated 11/15 times):** **B1** is the direct code-level cause, and **I1** is the concrete fix.

- **Run 15 duplicate names, run 16 duplicate/near-duplicate themes, and the all-model cross-call redundancy finding:** **B3** explains these best. The code asks for `"more"` twice but never communicates which names/themes are already used, and **I3** is the corresponding remedy.

- **Run 16 producing 20 levers, run 10 producing a malformed 16-item file, and the split 2-option + 1-option lever in `gta_game`:** **B2** is the root cause. The schema and merge path never enforce per-call count, per-lever option count, or final total count, so malformed-but-parseable outputs are written as successful artifacts.

- **Runs 14 and 15 drifting away from literal `Immediate:` / `Systemic:` / `Strategic:` labels, plus run 16 prefix leakage and bracket placeholders:** **B2** also explains these. Those prompt rules are never implemented as validators, so the runner has no way to reject or retry contract-broken outputs.

- **Run 09 failing all five plans due to an unavailable model alias:** **B4** is the operational explanation. The runner has no startup preflight, so it rediscovers the same invalid model name once per plan. **S2** suggests the worker-resolution path may also drift from runtime config selection.

- **Run 11 empty-JSON extraction failures and run 13 truncated wrapper-object failure:** the current code path does not have a targeted repair/retry stage for structured-output extraction problems. **S3** shows why those failures are only recorded as generic strings today, and **I5** is the clearest mitigation.

- **Run 12 being structurally excellent but overly verbose:** this is not a parsing bug, but **I7** points to the missing code-level control. The prompt enforces counts, not length, so verbose models can stay “correct” while still overspending tokens.

## Summary

- The strongest confirmed current bug is **B2**: the step’s contract lives mostly in prose, not in executable validation, so malformed outputs are accepted and written as successful results.
- The clearest prompt-construction bug is **B1**: the prompt contains the exact metric phrase that later leaks into outputs.
- The clearest multi-call logic defect is **B3**: follow-up turns say only `"more"`, which all but guarantees cross-call duplication on weaker models.
- On the runner side, **B4** is the main operational gap: invalid model names should fail once at startup, not once per plan.
- The highest-leverage fixes are **I2** (real validation), **I3** (stateful follow-up prompts), **I1** (remove the copied exemplar), and **I4** (model preflight).
