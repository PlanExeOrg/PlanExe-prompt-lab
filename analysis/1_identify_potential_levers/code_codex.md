# Code Review (codex)

## Bugs Found

- **B1 — Template leakage is baked into the shipped system prompt.** `worker_plan/worker_plan_internal/lever/identify_potential_levers.py:93-96` includes a literal example, `"Systemic: 25% faster scaling through..."`, inside the production prompt constant. Because this string is part of the actual prompt assembly, weaker models can copy it verbatim instead of abstracting the pattern. That directly matches the repeated `"25% faster scaling through"` phrasing observed in the history runs.

- **B2 — The step never enforces the promised cardinality contract, so malformed batches are accepted and flattened into final output.** The raw schema says levers are `list[Lever]` and only describes `"Propose exactly 5 levers."` without validating length at `worker_plan/worker_plan_internal/lever/identify_potential_levers.py:57-59`. The options field is also just `list[str]` with a loose description at `worker_plan/worker_plan_internal/lever/identify_potential_levers.py:33-35`, even though the prompt requires exactly 3 options at `worker_plan/worker_plan_internal/lever/identify_potential_levers.py:88-90`. Then `IdentifyPotentialLevers.execute()` blindly flattens every returned lever from every batch at `worker_plan/worker_plan_internal/lever/identify_potential_levers.py:203-219`, and `prompt_optimizer/runner.py:114-129` marks the plan `ok` as long as parsing did not throw. This is the code path that allowed run `16` to save 20 levers after one raw response returned 10 levers (`history/0/16_identify_potential_levers/outputs/20250321_silo/002-9-potential_levers_raw.json:126`, `history/0/16_identify_potential_levers/outputs/20250321_silo/002-9-potential_levers_raw.json:228`).

- **B3 — Exact output-shape requirements are prompt-only; the code does not validate them, so formally wrong outputs still succeed.** The prompt demands literal consequence markers, `%`-style measurable outcomes, exact review wording, and banned option prefixes at `worker_plan/worker_plan_internal/lever/identify_potential_levers.py:92-126`. But the only validator in the model is `parse_options()` for stringified arrays at `worker_plan/worker_plan_internal/lever/identify_potential_levers.py:40-51`. There is no validation for `consequences`, no validation for `review_lever`, and no validation that each option is a full strategy rather than a label. `prompt_optimizer/runner.py:123-129` therefore records these runs as success even when outputs violate the prompt contract. This explains why runs could be `ok` while still containing `Trade-off:` reviews, missing `Immediate:` / arrow syntax, or `Option 1:` leakage.

- **B4 — Worker metadata is resolved from every config file, not the active model profile, so the recorded `workers` value can be silently wrong.** `_resolve_workers()` merges every `llm_config/*.json` file into one map at `prompt_optimizer/runner.py:224-236` and picks `luigi_workers` from that merged dictionary at `prompt_optimizer/runner.py:239-254`. That bypasses the profile-selection logic used elsewhere in PlanExe (`PLANEXE_MODEL_PROFILE`, custom config filename, fallback behavior), so the runner can record a stale or inactive config entry if a model name exists in multiple profile files. This is a real configuration bug even though it does **not** explain the 20-lever case by itself.

## Suspect Patterns

- The prompt-lab runner is measuring an intermediate brainstorm stage, not the later deduplicated/vital-few deliverable. `IdentifyPotentialLevers.execute()` always performs exactly three chat turns via `user_prompt`, `"more"`, `"more"` at `worker_plan/worker_plan_internal/lever/identify_potential_levers.py:155-159`, and it intentionally flattens all three batches at `worker_plan/worker_plan_internal/lever/identify_potential_levers.py:203-206`. `prompt_optimizer/runner.py:118-121` then writes that intermediate output straight to `002-10-potential_levers.json`. In the full pipeline, later tasks do the reduction work (`worker_plan/worker_plan_internal/plan/run_plan_pipeline.py:395-532`). So the repeated 15-lever pattern is expected at this stage; the true bug is that one batch was allowed to return 10 instead of 5.

- Cross-call context is weak and model-dependent. The second and third turns are just `"more"` at `worker_plan/worker_plan_internal/lever/identify_potential_levers.py:155-159`, with no explicit instruction to avoid prior lever names. The previous assistant response is appended as `raw.model_dump()` at `worker_plan/worker_plan_internal/lever/identify_potential_levers.py:193-198`, which relies on adapter-specific serialization. That may be enough for strong models, but it gives weaker models very little structured pressure to produce non-overlapping levers.

- The raw schema and cleaned export intentionally use different field names: `review_lever` in the model at `worker_plan/worker_plan_internal/lever/identify_potential_levers.py:36-38`, but `review` in the cleaned output at `worker_plan/worker_plan_internal/lever/identify_potential_levers.py:81-83` and `worker_plan/worker_plan_internal/lever/identify_potential_levers.py:217`. This does **not** appear to be the direct cause of the run `13` parse failure, because the raw parser already expects `review_lever`; however, it does create schema drift between the prompt-facing raw format and the exported artifact.

## Improvement Opportunities

- **I1 — Add hard validators for the actual contract.** Enforce exactly 5 levers, exactly 3 options per lever, and minimum structural checks for `consequences` and `review_lever` inside `worker_plan/worker_plan_internal/lever/identify_potential_levers.py`. If a batch violates the contract, fail that batch before flattening and writing `002-10-potential_levers.json`.

- **I2 — Replace copyable numeric examples with placeholders.** Change the prompt text at `worker_plan/worker_plan_internal/lever/identify_potential_levers.py:93-96` from a concrete phrase (`25% faster scaling through`) to an abstract template such as `[N]% [measurable outcome] through [mechanism]`.

- **I3 — Make later turns explicitly non-overlapping.** Replace the bare `"more"` prompts at `worker_plan/worker_plan_internal/lever/identify_potential_levers.py:155-159` with something like “Generate 5 new levers that do not overlap with these prior names: ...”. Even a simple prior-name list would likely reduce exact-name and near-name duplicates.

- **I4 — Add batch-level recovery instead of all-or-nothing failure.** `IdentifyPotentialLevers.execute()` currently raises as soon as one `sllm.chat(...)` call fails at `worker_plan/worker_plan_internal/lever/identify_potential_levers.py:172-191`, and `prompt_optimizer/runner.py:130-138` converts that into a whole-plan failure. A better pattern would be to retry malformed structured output with validation feedback, salvage common near-valid cases, or at least retry only the failed batch instead of discarding the whole plan.

- **I5 — Resolve worker count from the selected profile, not all configs.** `prompt_optimizer/runner.py:224-254` should use the same profile-selection rules as the main application, or record the resolved config filename in `meta.json` so later analyses do not overfit on misleading `workers` metadata.

- **I6 — If prompt-lab is supposed to judge final 5-lever quality, add an option to run downstream reduction steps.** Right now `prompt_optimizer/runner.py:114-121` stops at `IdentifyPotentialLevers`. If evaluators care about the final “vital few,” the runner should optionally chain into the deduplication and selection stages instead of analyzing a deliberately over-complete intermediate artifact.

## Trace to Insight Findings

- The template leakage noted in `insight_claude.md` is directly explained by **B1**. The copied phrase is not accidental model behavior; it is present verbatim in the production prompt at `worker_plan/worker_plan_internal/lever/identify_potential_levers.py:95`.

- The run `16` lever inflation is explained by **B2**, but **not** by worker-count-dependent batching. The code always creates three turns (`worker_plan/worker_plan_internal/lever/identify_potential_levers.py:155-159`), and thread-pool workers only change plan-level concurrency (`prompt_optimizer/runner.py:375-389`). The history artifact shows what really happened: three responses with lever counts `[5, 5, 10]`, not four 5-lever batches (`history/0/16_identify_potential_levers/outputs/20250321_silo/002-9-potential_levers_raw.json:65`, `history/0/16_identify_potential_levers/outputs/20250321_silo/002-9-potential_levers_raw.json:126`, `history/0/16_identify_potential_levers/outputs/20250321_silo/002-9-potential_levers_raw.json:228`).

- The structural drift called out across runs in both insight files — wrong option counts, missing consequence markers, `Trade-off:` reviews, option-label leakage — is explained by **B3**. Those constraints exist only in prompt prose (`worker_plan/worker_plan_internal/lever/identify_potential_levers.py:92-126`), while the executable validation is effectively just type-checking (`worker_plan/worker_plan_internal/lever/identify_potential_levers.py:40-51`).

- The repeated lever names and semantic overlap across batches are most plausibly explained by the suspect multi-turn design plus **I3**. Later turns get only `"more"` and no explicit exclusion set (`worker_plan/worker_plan_internal/lever/identify_potential_levers.py:155-159`), so the model is not strongly constrained to stay distinct from its own prior batches.

- The hard failure in run `13` is better explained by brittle parse/recovery logic than by the `review_lever` field name. The raw schema already expects `review_lever` at `worker_plan/worker_plan_internal/lever/identify_potential_levers.py:36-38`, so the source does not support a `review_lever`-versus-`review` parser mismatch hypothesis. What the code *does* show is an all-or-nothing structured call at `worker_plan/worker_plan_internal/lever/identify_potential_levers.py:172-191`, with the runner turning any such failure into a plan error at `prompt_optimizer/runner.py:130-138`. That matches the observed `outputs.jsonl` failure record at `history/0/13_identify_potential_levers/outputs.jsonl:4`.

- The “15 levers instead of a 5-lever deliverable” complaint in `insight_codex.md` is partly an evaluation-stage mismatch, not purely a bug. The reviewed code intentionally emits three 5-lever batches at this step (`worker_plan/worker_plan_internal/lever/identify_potential_levers.py:155-159`, `worker_plan/worker_plan_internal/lever/identify_potential_levers.py:203-206`), and the full pipeline reduces later (`worker_plan/worker_plan_internal/plan/run_plan_pipeline.py:395-532`). The real defect is that no validator stopped a 10-lever batch from entering that intermediate artifact.

## Summary

- The strongest confirmed code issues are prompt leakage (**B1**), missing contract validation (**B2**, **B3**), and profile-blind worker resolution (**B4**).
- The key correction to the insight files is that run `16` did **not** create an extra batch because `workers=1`; it created three batches, and one malformed batch returned 10 levers.
- The run `13` failure looks like brittle structured-output recovery, not a raw `review_lever` schema mismatch in these two files.
- If only one change is made first, it should be **I1**: add real validators before flattening/saving. That would stop the 20-lever failure mode and convert many current “quality issues” into explicit, debuggable validation errors.
