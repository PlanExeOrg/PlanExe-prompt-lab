# Assessment: experiment: remove lever_index from Lever schema

## Issue Resolution

**What the PR was supposed to fix:** PR #361 removed `lever_index: int` from the `Lever` Pydantic
schema. The field was generated by the LLM but never transferred to `LeverCleaned` (which assigns a
fresh `lever_id = str(uuid.uuid4())`). The stated risk was that removing this "sink" field might
cause models to insert sequential index prefixes ("1.", "2.", "3.") into `name` or `consequences`
fields instead.

**Is the issue resolved?** Yes — fully. `lever_index` is absent in all after-run raw outputs.
Confirmed across four models:
- haiku (run 3/00) hong_kong_game `002-9-potential_levers_raw.json`: levers contain only `name`,
  `consequences`, `options`, `review_lever`; no `lever_index` field.
- gpt-5-nano (run 96), llama3.1 (run 94), qwen3 (run 97): same structure confirmed.

**Did the risk materialize?** No. Zero models inserted "1.", "2." index prefixes into `name` or
`consequences` fields across all 7 models × 5 plans. Verified by direct inspection of lever names:
haiku (run 3/00): "Narrative Subversion Strategy", "Director's Relationship to Hong Kong";
llama3.1 (run 94): "Cinematic DNA", "Location-Driven Production"; gpt-5-nano (run 96): "HK-based
financing architecture with pre-sales", "Casting and creative leadership"; qwen3 (run 97):
"Location-Driven Paranoia Architecture", "Talent Hybridization Strategy".

**Residual symptoms of the original issue?** None. The field is cleanly removed.

**Observed side effect:** Removing `lever_index: int` made each lever's JSON ~15–20 tokens
shorter. Haiku's per-call lever count shifted from ~6–7 to ~8–12, causing the step-gate
(`min_levers = 15`) to fire after 2 calls for ALL 5 haiku plans (up from 2/5 before).
This is not a symptom of the stated issue but is an unintended behavioral consequence.

### Evidence for Regressions

**Haiku call efficiency (-20pp):** Confirmed via events.jsonl. Run 93 (before): 2 partial_recovery
events — gta_game 2/3, silo 2/3. Run 3/00 (after): 5 partial_recovery events — ALL 5 plans exit
after 2 calls (gta_game, hong_kong_game, sovereign_identity, silo, parasomnia). Source: direct
inspection of both events.jsonl files.

**Mechanism:** Haiku tokens per call remain similar (~2600–2900/call before and after). With lever_index
removed, each lever's JSON is ~15–20 tokens shorter, allowing haiku to fit 8–12 levers per call
instead of 6–7. Two calls × 8+ levers = 16+ ≥ min_levers (15), triggering early step-gate exit
every time. Before, only 2/5 plans had this property.

**Important caveat on "partial_recovery" metrics:** The code review (code_claude B1) confirms that
`runner.py:517–523` fires `partial_recovery` whenever `calls_succeeded < 3`, regardless of whether
`min_levers` was met. A haiku run producing 24 levers in 2 calls is a healthy early exit, not a
failure. The −20pp "regression" in haiku call efficiency is therefore measuring a behavioral shift
(over-generation → fewer calls needed), not genuine failures. All 35 plans completed with
`status: "ok"`.

**llama3.1 review template lock (NEW FINDING, NOT IN INSIGHT FILES):** Direct inspection of run 87
(before) vs. run 94 (after) hong_kong_game `002-10-potential_levers.json` reveals:
- Before (run 87): Reviews use "Not X could result in Y" pattern — e.g., "Not leveraging Hong Kong's
  existing infrastructure could lead to missed opportunities…", "Not selecting the right director
  could result in a film that fails to capture…" (pre-existing lock, different pattern)
- After (run 94): Reviews use "This lever introduces the trade-off between X and Y, leaving
  unaddressed Z" pattern — ALL 6 reviews sampled follow this exact structure:
  - "This lever introduces the primary trade-off of balancing cultural authenticity with commercial
    appeal, leaving unaddressed the challenge of conveying the city's unique paranoia machine…"
  - "This lever introduces the trade-off between financial efficiency and artistic expression,
    leaving unaddressed the need for a clear vision…"
  - "This lever introduces the trade-off between artistic vision and social responsibility, leaving
    unaddressed the challenge of creating a narrative…"

The phrase "leaving unaddressed" is taken verbatim from the PR #358 field description: "state the
specific gap the three options leave unaddressed." This secondary template lock was already present
in haiku after PR #358 (~85% in run 93); run 94 shows it has now migrated to llama3.1 as well.
Whether this migration was triggered by PR #361 (schema compactness change altering context
window balance) or is stochastic variance (llama3.1 is a weak sequential model) is unconfirmed.
A re-run of llama3.1 with lever_index restored would disambiguate.

**llama3.1 option content length (NEW FINDING, NOT IN INSIGHT FILES):** Run 87 (before) options
are full descriptive sentences:
- "Utilize the city's existing infrastructure to film in public spaces during off-peak hours,
  minimizing disruptions." (~100 chars)
- "Select a Hong Kong or Asian filmmaker with a proven track record in thrillers or noirs to bring
  a nuanced understanding of the city's rhythms and cinematic language." (~160 chars)

Run 94 (after) options are 3–7 word noun phrases:
- "Preserve a distinct Western tone" (~32 chars)
- "Integrate local and Western styles organically" (~46 chars)
- "Maintain a tight budget and schedule" (~36 chars)

These are essentially labels, not actionable strategies. This is a severe content quality regression
for llama3.1 that the insight files did not check. The cause is unknown: possibly related to
lever_index removal (shorter JSON per lever → different generation patterns), possibly stochastic.

**"None/all three options" lock — haiku (pre-existing):** Confirmed at ~87% in run 3/00
hong_kong_game (14/16 reviews). Direct inspection of run 3/00 `002-10-potential_levers.json`:
all sampled reviews follow "The trade-off is between X and Y; none of the three options [directly]
address Z." Pattern: "The trade-off is between honoring the source material and achieving creative
distinctiveness; all three options leave unaddressed the risk that any ending may feel redundant…"
This lock is pre-existing from PR #358 and unchanged by PR #361.

**Baseline context:** Baseline training data (hong_kong_game) uses "Controls X vs. Y. Weakness: Z"
review format and "Immediate → Systemic → Strategic" consequence chain markers. Neither format
appears in before or after batches — this is a pre-existing deviation predating this analysis.
Baseline consequences also contain fabricated percentages ("15% higher audience engagement",
"30% increase in streaming revenue") — the current system prompt (Section 5) correctly prohibits
these; both before and after batches show improvement over baseline on this dimension.

## New Issues

### N1 — Haiku consistently triggers step-gate after 2 calls (ALL 5 plans)
After PR #361, haiku generates ~8–12 levers per call (up from ~6–7), crossing `min_levers = 15`
in 2 calls for every plan. The third call uses a uniquely constrained prompt ("Generate MORE levers
with completely different names") that adds orthogonal perspectives. Losing this call for all 5
haiku plans reduces per-plan lever diversity. Fix: raise `min_levers` from 15 to 18
(synthesis Direction 1 / code_claude I1).

### N2 — llama3.1 option content quality severely regressed (run 94)
llama3.1 hong_kong_game options went from full actionable sentences to 3–7 word noun phrases.
This degrades the usefulness of every lever in the downstream pipeline. Cause is unconfirmed
(lever_index removal vs. stochastic variance). Must be verified in the next iteration: if llama3.1
options remain degraded in run 95+ with the min_levers fix in place, a separate PR targeting
llama3.1 output quality is needed.

### N3 — llama3.1 review template lock migrated ("leaving unaddressed")
The secondary template lock from PR #358's field description ("state the specific gap the three
options leave unaddressed") has now appeared in llama3.1 (run 94) in addition to haiku. The lock
affects the review quality for both models. Whether triggered by PR #361 or by PR #358 with delayed
manifestation is unconfirmed. This strengthens the urgency of fixing the `review_lever` field
description (the pre-existing Direction 3 target from analysis 40).

### N4 — OPTIMIZE_INSTRUCTIONS missing schema-compactness note
The current `OPTIMIZE_INSTRUCTIONS` states "Over-generation is fine; step 2 handles extras." This
is true at the plan level but misleading about per-call over-generation effects on step-gate timing.
It also does not document the confirmed causal chain: fewer Lever schema fields → more compact JSON
per lever → more levers fit per call → earlier step-gate exit. Proposed additions (code_claude I4,
I5) are low-risk and should be bundled with the next PR.

## Verdict

**CONDITIONAL**: The schema cleanup is correct and should be kept — `lever_index` was confirmed dead
code and its removal did not introduce the predicted risk (index prefix leakage). Two conditions must
be addressed: (1) raise `min_levers` from 15 to 18 to restore 3-call behavior for haiku (N1 above),
and (2) verify llama3.1 option quality in the next iteration to confirm whether N2 is stochastic
or structurally caused by the schema change.

## Recommended Next Change

**Proposal:** Raise `min_levers` from 15 to 18 in `identify_potential_levers.py:285`. This is a
single integer constant change with no prompt or downstream schema impact.

**Evidence:** The synthesis (analysis 41) and code review both independently recommend this change.
The causal chain is confirmed: removing lever_index → more compact per-lever JSON (~15–20 fewer
tokens per lever) → haiku fits 8–12 levers per call → 2 × 8 = 16 ≥ min_levers (15) → step-gate
fires after 2 calls for ALL 5 plans. Before PR #361, haiku needed 3 calls for 3/5 plans.
Raising `min_levers` to 18 forces a third call even at 8 levers/call (2 × 8 = 16 < 18).
Models generating 6–7/call already need 3 calls (3 × 7 = 21 ≥ 18) and are unaffected.

**Verify in next iteration:**
- **Primary check:** haiku (run 3/01 or next) partial_recovery drops from 5/5 plans to ≤2/5.
  Confirm via events.jsonl: expect at most 2 partial_recovery events for haiku.
- **Secondary check:** 5-model efficiency (gpt-oss, gpt-5-nano, qwen3, gpt-4o-mini, gemini)
  remains 100% — they should not be affected by raising the threshold.
- **llama3.1 check:** Verify whether llama3.1 option quality (run 94 severe regression: 3-7 word
  phrases) recovers in the next run. If llama3.1 options remain degraded after the min_levers fix,
  the regression is structural (lever_index removal caused it) rather than stochastic. Watch
  specifically: are option lengths back to full sentences (~100–300 chars), or still noun phrases
  (~20–50 chars)?
- **Template lock check:** Confirm the "none/all three options" review lock remains ~87% for haiku
  (it will not be fixed by this change — that requires a separate field description PR, which is
  Direction 3 in the synthesis and the top priority from analysis 40).
- **Over-generation check:** Confirm haiku generates ~6–7 levers/call after the fix forces a third
  call (expected: haiku continues generating 8–12/call but now completes 3 calls, yielding 24–36
  levers total — well above min_levers = 18 after call 3). The step-gate should exit on call 3
  rather than call 2 for most plans.

**What could go wrong:**
- Haiku generating ≥ 18 levers in 2 calls (≥ 9/call) would still exit after 2 calls. This is the
  upper edge of observed haiku behavior (silo: ~12/call); gta_game and hong_kong_game were at ~8/call.
  If silo and gta_game stay at 12+/call, they may still exit early. Acceptable — the fix helps 3+
  of 5 plans.
- If llama3.1 option quality regression is structural (caused by lever_index removal altering
  llama3.1's generation pattern), raising min_levers will not fix it. A separate intervention
  targeting llama3.1 output quality would be needed.
- Models at exactly 6/call: 3 × 6 = 18 ≥ 18 on the 3rd call — they exit on the 3rd call, same as
  before. No extra calls needed. Safe.

**Prerequisite issues:** None. The PR is a one-line code change with no dependencies.

**Note on iteration order:** The synthesis recommends raising min_levers BEFORE fixing the
`review_lever` field description. This is correct: fixing the step-gate regression first gives
haiku a proper 3-call sample in the next analysis, making the template-lock fix's effect more
measurable. If template lock and step-gate were fixed simultaneously, it would be harder to isolate
which change improved haiku's per-plan diversity.
