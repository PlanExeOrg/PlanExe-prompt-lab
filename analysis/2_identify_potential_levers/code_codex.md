# Code Review (codex)

## Bugs Found

- **B1 — The shipped system prompt contains the exact exemplar strings that later leak into model output.** The prompt hard-codes both `"Systemic: 25% faster scaling through..."` and `"Material Adaptation Strategy"` as examples. Those are not abstract instructions; they are concrete output fragments, so weaker models can copy them verbatim into consequences and lever names. This directly matches the insight reports' fake-percentage reuse and template-name leakage. Refs: `/Users/neoneye/git/PlanExeGroup/PlanExe/worker_plan/worker_plan_internal/lever/identify_potential_levers.py:93-105`.

- **B2 — The multi-call loop asks only for `"more"`, so call 2 and call 3 are not actually constrained to produce new levers.** `IdentifyPotentialLevers.execute()` sends the original prompt once, then follows with two bare `"more"` turns while carrying forward the full prior chat history. It never summarizes already-used lever names, never says `do not repeat`, and never asks for a different slice of the solution space. That is a concrete chat-history-management bug: the runner is doing three calls, but only the first call is properly specified. This explains repeated five-name sets, low cross-call diversity, and some conversational drift on weaker models. Refs: `/Users/neoneye/git/PlanExeGroup/PlanExe/worker_plan/worker_plan_internal/lever/identify_potential_levers.py:155-200`.

- **B3 — The step's output contract is prose-only; the code never enforces it before writing `ok` artifacts.** The prompt requires exactly 5 levers per response, exactly 3 options per lever, exact chain labels, and exact review phrasing. But the schema uses unconstrained `list[Lever]` and `list[str]`, and the only validator merely parses stringified arrays. After that, all returned levers are blindly flattened and written to `002-10-potential_levers.json`, and the runner marks the plan `ok` without any post-merge validation. This is the root cause of 16/19-lever files, missing `Immediate:` labels, review-prefix drift, and other malformed-but-parseable outputs surviving as successes. Refs: `/Users/neoneye/git/PlanExeGroup/PlanExe/worker_plan/worker_plan_internal/lever/identify_potential_levers.py:33-51`, `/Users/neoneye/git/PlanExeGroup/PlanExe/worker_plan/worker_plan_internal/lever/identify_potential_levers.py:57-59`, `/Users/neoneye/git/PlanExeGroup/PlanExe/worker_plan/worker_plan_internal/lever/identify_potential_levers.py:78-82`, `/Users/neoneye/git/PlanExeGroup/PlanExe/worker_plan/worker_plan_internal/lever/identify_potential_levers.py:203-219`, `/Users/neoneye/git/PlanExeGroup/PlanExe/worker_plan/worker_plan_internal/lever/identify_potential_levers.py:257-259`, `/Users/neoneye/git/PlanExeGroup/PlanExe/prompt_optimizer/runner.py:118-129`.

- **B4 — The prompt-optimizer runner has no repair path for malformed structured output, even though the executor supports it.** `run_single_plan()` creates `LLMExecutor` with default settings, so validation retries stay at `0`. `IdentifyPotentialLevers.execute()` also never checks `llm_executor.validation_feedback` before retrying a bad response; it just wraps any exception and fails the plan. With a single requested model, one truncated JSON or one conversational non-JSON reply becomes a terminal plan error. That matches the weak-model failure pattern in the insight files. Refs: `/Users/neoneye/git/PlanExeGroup/PlanExe/prompt_optimizer/runner.py:93-116`, `/Users/neoneye/git/PlanExeGroup/PlanExe/prompt_optimizer/runner.py:130-137`, `/Users/neoneye/git/PlanExeGroup/PlanExe/worker_plan/worker_plan_internal/lever/identify_potential_levers.py:172-191`, `/Users/neoneye/git/PlanExeGroup/PlanExe/worker_plan/worker_plan_internal/llm_util/llm_executor.py:166-180`, `/Users/neoneye/git/PlanExeGroup/PlanExe/worker_plan/worker_plan_internal/llm_util/llm_executor.py:193-207`.

## Suspect Patterns

- **S1 — The continuation context grows with full prior assistant payloads, which likely amplifies verbosity and timeout risk on later calls.** After each response, the code appends the assistant's entire structured output back into `chat_message_list`. By call 3, the model sees the system prompt, the full original plan prompt, two prior JSON-rich assistant replies, and two `"more"` nudges. That context-growth pattern is a plausible explanation for the insight reports where stronger models become much longer on later calls and eventually time out. Refs: `/Users/neoneye/git/PlanExeGroup/PlanExe/worker_plan/worker_plan_internal/lever/identify_potential_levers.py:163-197`.

- **S2 — The optimizer appears to score an intentionally pre-dedup artifact.** The step file says duplicates are expected and are cleaned up by `deduplicate_levers.py`, but `prompt_optimizer.runner` stops after writing `002-10-potential_levers.json`. If prompt-lab is evaluating that merged file as the main quality artifact, it is measuring a stage that the pipeline itself documents as duplicate-prone. This likely explains why baseline and several runs all share the same repeated-five-name pattern. Refs: `/Users/neoneye/git/PlanExeGroup/PlanExe/worker_plan/worker_plan_internal/lever/identify_potential_levers.py:4-5`, `/Users/neoneye/git/PlanExeGroup/PlanExe/worker_plan/worker_plan_internal/lever/identify_potential_levers.py:203-219`, `/Users/neoneye/git/PlanExeGroup/PlanExe/prompt_optimizer/runner.py:114-121`.

- **S3 — `_resolve_workers()` does not use the same config-selection path as actual model creation.** The runner computes parallelism by globbing every `llm_config/*.json`, while `get_llm()` loads the active profile via `PlanExeLLMConfig.load(model_profile=...)`. That means worker planning can silently disagree with the actual model profile being used, especially for custom profiles. I do not see a direct match to the current insight failures, but it is a real configuration-drift risk. Refs: `/Users/neoneye/git/PlanExeGroup/PlanExe/prompt_optimizer/runner.py:224-254`, `/Users/neoneye/git/PlanExeGroup/PlanExe/worker_plan/worker_plan_internal/llm_factory.py:53-55`, `/Users/neoneye/git/PlanExeGroup/PlanExe/worker_plan/worker_plan_internal/llm_factory.py:185-199`.

## Improvement Opportunities

- **I1 — Remove content-like exemplars from the system prompt.** Replace `"Material Adaptation Strategy"` and `"25% faster scaling through..."` with neutral placeholders or meta-instructions, and explicitly say `do not reuse examples verbatim`. This is the cleanest fix for template leakage and fake percentage parroting. Refs: `/Users/neoneye/git/PlanExeGroup/PlanExe/worker_plan/worker_plan_internal/lever/identify_potential_levers.py:93-105`.

- **I2 — Make follow-up turns novelty-aware instead of sending `"more"`.** Before call 2 and call 3, extract prior lever names/themes and inject a follow-up user message such as `You already generated: [...]. Generate 5 different levers with different names and different core tensions.` This uses information the code already has and directly targets the cross-call duplication seen in the insights. Refs: `/Users/neoneye/git/PlanExeGroup/PlanExe/worker_plan/worker_plan_internal/lever/identify_potential_levers.py:155-206`.

- **I3 — Enforce the prompt contract in code, not just prose.** Add validators or a post-merge gate for: 5 levers per call, 3 options per lever, exact `Immediate:` / `Systemic:` / `Strategic:` prefixes, exact `Controls ... vs. .... Weakness:` review form, and a hard merged-count cap. If validation fails, retry or mark the plan as `error` instead of saving malformed artifacts as `ok`. Refs: `/Users/neoneye/git/PlanExeGroup/PlanExe/worker_plan/worker_plan_internal/lever/identify_potential_levers.py:23-82`, `/Users/neoneye/git/PlanExeGroup/PlanExe/worker_plan/worker_plan_internal/lever/identify_potential_levers.py:203-219`, `/Users/neoneye/git/PlanExeGroup/PlanExe/prompt_optimizer/runner.py:118-129`.

- **I4 — Use the executor's validation-retry hooks for weak-model recovery.** Instantiate `LLMExecutor` with non-zero `max_validation_retries`, detect `validation_feedback`, and prepend a compact repair instruction like `Your previous reply was invalid JSON / wrong schema; return one valid JSON object only.` That would give truncated or chatty models one or two self-correction chances before the whole plan fails. Refs: `/Users/neoneye/git/PlanExeGroup/PlanExe/prompt_optimizer/runner.py:93-116`, `/Users/neoneye/git/PlanExeGroup/PlanExe/worker_plan/worker_plan_internal/lever/identify_potential_levers.py:172-183`, `/Users/neoneye/git/PlanExeGroup/PlanExe/worker_plan/worker_plan_internal/llm_util/llm_executor.py:170-180`, `/Users/neoneye/git/PlanExeGroup/PlanExe/worker_plan/worker_plan_internal/llm_util/llm_executor.py:193-207`.

- **I5 — Compress continuation context or set explicit brevity budgets.** Instead of replaying full prior assistant payloads, pass a compact summary of prior lever names plus a length budget for each field. That should reduce the call-3 bloat / timeout pattern seen on strong models while keeping the diversity benefit of multi-call generation. Refs: `/Users/neoneye/git/PlanExeGroup/PlanExe/worker_plan/worker_plan_internal/lever/identify_potential_levers.py:163-197`.

- **I6 — Align prompt-lab scoring with the pipeline boundary that downstream steps actually consume.** Either run `deduplicate_levers` inside the optimizer harness, or save both pre-dedup and post-dedup artifacts and score them separately. That would distinguish `generation breadth` problems from `expected intermediate duplication`. Refs: `/Users/neoneye/git/PlanExeGroup/PlanExe/worker_plan/worker_plan_internal/lever/identify_potential_levers.py:4-5`, `/Users/neoneye/git/PlanExeGroup/PlanExe/prompt_optimizer/runner.py:114-121`.

## Trace to Insight Findings

- **Template leakage and hallucinated numbers:** **B1** explains the repeated `Material Adaptation Strategy` copies and the recurring `25%`-style systemic claims described in both insight files.

- **Cross-call duplication / repeated five-name sets:** **B2** is the strongest code-level explanation. The current continuation turns do not ask for novelty, so repetition is the default behavior unless a model independently volunteers diversity.

- **Overproduced files and literal-format drift:** **B3** explains the 16-lever / 19-lever artifacts, missing `Immediate:` openers, and `Trade-off:` / missing `Weakness:` review drift. The code never validates these requirements before writing success artifacts.

- **Weak-model reliability failures:** **B4** explains why truncated JSON and conversational replies become terminal plan failures. The code uses structured output, but it does not enable the executor's schema-repair path or any targeted retry logic.

- **Haiku verbosity growth and timeout:** **S1** plausibly explains the pattern where later calls get richer and much longer. The full prior assistant payload is repeatedly fed back into the next call, which increases context size and likely encourages longer continuations.

- **Baseline and many runs sharing the same duplicate-heavy merged artifact:** **S2** explains why that pattern persists across runs. The optimizer is inspecting the pre-dedup stage that the step itself documents as duplicate-prone.

## Summary

- The clearest confirmed defect is **B3**: the step's contract is mostly written in English, not enforced in code, so invalid artifacts are still saved as successes.
- The clearest prompt-construction defects are **B1**'s exemplar leaks: the system prompt contains the exact strings that later show up as copied output.
- The clearest multi-call logic bug is **B2**: follow-up turns say only `"more"`, so the second and third calls are not actually directed toward novelty.
- For reliability, **B4** is the main hardening gap: malformed structured output gets one shot and then the whole plan fails.
- The highest-leverage fixes are **I2** (novelty-aware follow-up prompts), **I3** (real validation), **I1** (remove exemplar leakage), and **I4** (schema-repair retries). If prompt-lab wants to score downstream usefulness rather than raw generation breadth, **I6** is also important.
