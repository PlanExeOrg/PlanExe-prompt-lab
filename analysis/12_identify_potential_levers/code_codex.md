## PR Review

- The actual PR change is tiny and correctly scoped: it removes `max_length=7` from `DocumentDetails.levers` in `/Users/neoneye/git/PlanExeGroup/PlanExe/worker_plan/worker_plan_internal/lever/identify_potential_levers.py:81`. That does exactly what the PR description claims — it prevents a single 8-lever response from failing the whole call.

- I do **not** see a prompt-construction regression in `runner.py`. `load_user_prompt()` matches the real pipeline task’s prompt assembly in `/Users/neoneye/git/PlanExeGroup/PlanExe/prompt_optimizer/runner.py:56` and `/Users/neoneye/git/PlanExeGroup/PlanExe/worker_plan/worker_plan_internal/plan/run_plan_pipeline.py:380`, and `IdentifyPotentialLevers.execute()` now sends one system + one user message per call in `/Users/neoneye/git/PlanExeGroup/PlanExe/worker_plan/worker_plan_internal/lever/identify_potential_levers.py:211`. So the earlier doubled-user-prompt class of bug is not present here.

- The PR **does** fix the targeted failure mode, but it leaves an important gap: after removing the hard cap, the code has no replacement control for overflow. That means the system prompt’s “5 to 7 levers” rule is now advisory only, with no clamp, warning, or telemetry when a model returns 8 or 9. This is a follow-up bug, not a reason to restore the old hard failure.

- The PR also does not address the other failure families in the insight files because those live elsewhere in the control flow: malformed-but-parseable levers are still accepted, and extraction failures still lose their raw artifacts. So “keep the PR, then add soft overflow handling and stronger validation” is the right conclusion.

## Summary

- The PR under review fixes the exact `max_length=7` brittleness it targeted.
- The strongest remaining code bug is missing enforcement of the per-lever contract, which is why malformed 2-option levers and incomplete review strings still appear in successful outputs.
- The main PR gap is that overflow is now silently accepted instead of being softly trimmed or logged.
- The runner also throws away the most useful failure evidence, which makes the JSON-extraction failures much harder to debug than they need to be.
