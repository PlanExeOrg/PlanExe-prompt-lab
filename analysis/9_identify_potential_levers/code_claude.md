## Bugs Found

### B1 — Schema/code allow 5–7 levers; prompt says "EXACTLY 5"; follow-up call text says "5 to 7 MORE"

**Files:** `identify_potential_levers.py:78-82`, `identify_potential_levers.py:128`, `identify_potential_levers.py:203`

Three places control the per-call lever count, and they are out of sync:

| Location | Says |
|---|---|
| `DocumentDetails.levers` schema (line 78–82) | `min_length=5, max_length=7`, description "Propose 5 to 7 levers" |
| System prompt section 1 (line 128) | "You must generate 5 to 7 levers per response" |
| Follow-up call text (line 203) | `"Generate 5 to 7 MORE levers …"` |
| Registered `prompt_1` used in runs 67–73 | **"EXACTLY 5 levers per response"** |

When the runner passes the registered `prompt_1` as `system_prompt`, the pydantic schema still validates 6 or 7 levers as acceptable. The follow-up call text actively overrides the registered prompt's "EXACTLY 5" contract with "5 to 7 MORE". As a result, a call that returns 7 levers is never caught — it passes validation and lands directly in the cleaned output.

With 3 calls × up to 7 levers each, up to 21 levers can appear in a final output that is supposed to contain exactly 15. Removing the upper-bound mismatch (or enforcing the exact-5 contract through the schema) would eliminate this over-generation path.

### B2 — Merged lever list has no deduplication step

**File:** `identify_potential_levers.py:244-260`

After collecting all three responses, the code flatly extends a list:

```python
levers_raw: list[Lever] = []
for response in responses:
    levers_raw.extend(response.levers)
```

`generated_lever_names` (lines 195, 201, 240) is accumulated and injected into subsequent calls as a soft exclusion list, but it has no effect once the three calls are complete — the final flatten does nothing to remove duplicates that slipped through. If a model ignores the name exclusion (as llama3.1 does), the duplicate names survive unchanged into `002-10-potential_levers.json`.

The deduplication the module-level docstring (lines 4–9) mentions as a downstream step (`deduplicate_levers.py`) is external to this file and is not invoked by the runner, so duplicates created here are present in the data that analysis examines.

### B3 — No validation of individual lever fields before saving

**File:** `identify_potential_levers.py:249-260`

The clean-up step copies raw fields into `LeverCleaned` with zero field-level validation:

```python
for i, lever in enumerate(levers_raw, start=1):
    lever_id = str(uuid.uuid4())
    lever_cleaned = LeverCleaned(
        lever_id=lever_id,
        name=lever.name,
        consequences=lever.consequences,
        options=lever.options,
        review=lever.review_lever,
    )
    levers_cleaned.append(lever_cleaned)
```

Nothing checks:
- Whether `consequences` contains the review markers `"Controls"` or `"Weakness:"` that belong exclusively in `review_lever` (despite the explicit field description at lines 42–43 and 106–107).
- Whether `review_lever` contains both required sentences (the "Controls [A] vs. [B]." sentence and the "Weakness: …" sentence).
- Whether any field still holds bracket placeholders such as `[Invalidate existing power structures]`.

All three failure modes appear in the run artifacts and survive into the cleaned output files that subsequent analysis steps trust.

### B4 — `set_usage_metrics_path` called outside the lock in runner.py

**File:** `runner.py:106-109`, `runner.py:139-142`

When `workers > 1`, `run_single_plan` is invoked from multiple threads. The setup sequence is:

```python
set_usage_metrics_path(plan_output_dir / "usage_metrics.jsonl")   # line 106 — no lock

with _file_lock:                                                    # line 108
    dispatcher.add_event_handler(track_activity)
```

`set_usage_metrics_path` modifies global state without holding `_file_lock`. Thread A can set the global metrics path and then be pre-empted before its LLM calls begin; Thread B sets a different path; Thread A's LLM calls then write usage data to Thread B's path (or vice-versa). The teardown in `finally` has the same asymmetry:

```python
set_usage_metrics_path(None)           # line 140 — no lock
with _file_lock:                        # line 141
    dispatcher.event_handlers.remove(track_activity)
```

The severity depends on whether `set_usage_metrics_path` uses thread-local storage internally (which would make this harmless). If it stores a module-level variable, usage metrics from concurrent plan runs can be silently interleaved or written to the wrong file.

## Suspect Patterns

### S1 — `lever_index` is generated by the LLM but immediately discarded

**File:** `identify_potential_levers.py:28-30`, `identify_potential_levers.py:251-260`

`Lever.lever_index: int` is a required field that the LLM must produce. It is never transferred to `LeverCleaned`; the clean step assigns a sequential index via `enumerate(levers_raw, start=1)` and ignores the LLM's value. This wastes model output budget on a field that is thrown away. It may also subtly affect model behavior: a model generating call 2 thinks it is producing "levers 6–10" and may follow a numbering-induced pattern.

### S2 — Cross-call context is names-only; theme/content overlap goes undetected

**File:** `identify_potential_levers.py:195-215`

The anti-duplication mechanism passes only a flat list of previously generated lever names to subsequent calls (line 201–205). There is no mechanism to:

- Prevent content overlap across calls (similar options, reused consequences language, redundant themes).
- Detect that a follow-up call produced a lever that is semantically equivalent to a prior one under a different name.

This is the architectural reason haiku (run 73) under-generates on some calls: with fresh context, the model occasionally concludes there are no more genuinely novel angles and produces fewer than 5 levers rather than repeating prior content.

### S3 — Follow-up call appends `user_prompt` again after the exclusion list

**File:** `identify_potential_levers.py:202-206`

```python
prompt_content = (
    f"Generate 5 to 7 MORE levers with completely different names. "
    f"Do NOT reuse any of these already-generated names: [{names_list}]\n\n"
    f"{user_prompt}"
)
```

The full plan document (`user_prompt`) is appended after the names exclusion instruction. For a model that processes the message top-to-bottom, the plan content may functionally override the exclusion instruction — especially if the plan document is long and the instruction ends up as a small preamble. A safer layout would put the exclusion list at the bottom, closest to the schema enforcement boundary.

## Improvement Opportunities

### I1 — Align schema and follow-up text to the registered prompt's exact-5 contract

**File:** `identify_potential_levers.py:78-82`, `identify_potential_levers.py:203`

Change `DocumentDetails.levers` to:

```python
levers: list[Lever] = Field(
    min_length=5,
    max_length=5,
    description="Propose exactly 5 levers."
)
```

And change the follow-up call text to `"Generate 5 MORE levers"`. This closes the schema/prompt contract gap (B1) and makes over-generation detectable at parse time rather than silently accepted.

### I2 — Add post-merge field validation before saving

**File:** `identify_potential_levers.py:244-260`

Before appending each `LeverCleaned` item, validate:

1. `consequences` does not contain `"Controls"` or `"Weakness:"` (review contamination — B3).
2. `review_lever` contains both `"Controls"` and `"Weakness:"` (partial review format — B3).
3. No field contains `[` followed by a noun followed by `]` (placeholder check — B3).
4. `name` is not already present in the accumulated name set (duplicate check — B2).

Flagged levers could be logged as warnings or, in a retry-enabled runner, trigger a re-call for that batch.

### I3 — Include `strategic_rationale` in the clean output per call

**File:** `identify_potential_levers.py:275-300`

`DocumentDetails.strategic_rationale` (line 74–77) is generated by the LLM for every call and preserved in the raw JSON, but `save_clean` / `lever_item_list()` only serialises `self.levers`. Downstream pipeline steps that operate only on the clean file lose the per-call rationale. Emitting the rationale (e.g., as a top-level `"call_rationales"` list) in the clean output would make that reasoning available without requiring raw-file parsing.

### I4 — Remove the unused `lever_index` field from `Lever`

**File:** `identify_potential_levers.py:28-30`

Since `lever_index` is overwritten by `enumerate` in the clean step and never read, removing it from `Lever` reduces the token budget spent on a throwaway field and eliminates a potential source of confusing numbering behavior in follow-up calls (S1).

### I5 — Move `set_usage_metrics_path` inside the lock in runner.py

**File:** `runner.py:106-109`, `runner.py:139-142`

Group both the global-state writes inside `_file_lock` so they are atomic relative to each other:

```python
with _file_lock:
    set_usage_metrics_path(plan_output_dir / "usage_metrics.jsonl")
    dispatcher.add_event_handler(track_activity)
```

And similarly in `finally`. This closes the race window described in B4.

### I6 — Capture error type in PlanResult for finer failure analysis

**File:** `runner.py:130-138`

All exceptions (JSON parse failure, schema validation failure, LLM API error, network timeout) are collapsed into a single `error: str` field. Adding an `error_type` field — at minimum distinguishing `json_parse`, `schema_validation`, `llm_api`, and `unknown` — would let the analysis pipeline separate structural model failures (nemotron: JSON extraction) from schema failures (haiku: `List should have at least 5 items`) from transient errors, without requiring string-matching on the error message.

## Summary

**Most critical (affects data quality in every run):**

- **B1** is the highest-impact bug: the 5-vs-7 mismatch between the schema and the registered prompt means the code silently accepts 5–7 levers per call (and explicitly requests 7 in follow-up calls) even when the operational contract says "EXACTLY 5". This directly explains the 17–19 lever cleaned files in runs 68 and 73.

- **B3** is the second most impactful: the flat copy from `Lever` to `LeverCleaned` with no field-level checks means contaminated outputs (review text in consequences, placeholders, partial review fields) are indistinguishable from clean ones in `002-10-potential_levers.json`. Sixty of seventy-five levers in run 71 are affected.

**Significant but scoped:**

- **B2** (no merge-time dedup) only surfaces when a model ignores the name exclusion list; llama3.1 demonstrates it but stronger models do not.

- **B4** (thread-safety in runner.py) is latent: it only matters when `workers > 1` and depends on whether `set_usage_metrics_path` uses thread-local storage.

**Architectural note:**

The module-level docstring (lines 4–9) explicitly acknowledges that over-generation is expected and deduplication is deferred to `deduplicate_levers.py`. This design choice means **B1** and **B2** are intentional trade-offs rather than pure oversights — but the mismatch between that design and the registered prompt's "EXACTLY 5" constraint is unresolved and creates confusing analysis results. Aligning all three sites (schema, source system prompt, follow-up call text) to the same number would make the intent unambiguous.
