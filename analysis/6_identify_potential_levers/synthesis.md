# Synthesis

## Cross-Agent Agreement

Both insight agents and both code-review agents converge on the following points:

1. **No schema enforcement for lever count or summary presence.** `DocumentDetails.levers` is an unconstrained `list[Lever]` and `summary` is `Optional[str] = None`. Both agents independently call this out as the root cause of the 16-lever artifact (run 52) and null-summary regressions (14 null in run 47, 10 in run 48, 4 in run 51).

2. **The qwen3-30b contamination prohibition is already in the source.** Lines 38–39 of `identify_potential_levers.py` already say: "Do NOT include 'Controls ... vs.', 'Weakness:', or other review/critique text in this field — those belong exclusively in review_lever." Run 50 still shows 100% contamination. Both agents conclude that a description-only prohibition is insufficient; enforcement must happen after generation.

3. **llama3.1 option quality is degrading.** Average option length has dropped from ~88 chars (batch 4) to ~18 chars (batch 6). Both agents attribute this to the qualitative-only instruction ("not a label") and a possible model output-budget constraint. Both recommend a quantitative minimum.

4. **nemotron-3-nano has failed five consecutive full batches.** Zero parseable outputs across runs 24, 25, 32, 39, and 46. Both agents recommend skipping it.

5. **run 49 (gpt-5-nano) quality is high but verbosity is extreme.** 33,741 output tokens for one plan; both agents flag timeout risk and recommend a cap.

6. **Baseline is contaminated.** The baseline `consequences` fields contain review-text leakage, consistent with being generated by qwen3-30b. Both agents note this makes baseline-similarity an unreliable quality signal.

## Cross-Agent Disagreements

### Disagreement 1: What causes qwen3 contamination — missing instruction vs. chat structure?

- **Claude code review (S3):** The fallback `model_dump_json()` serialization in the assistant turn may confuse subsequent calls for conversation-conditioned models like qwen3.
- **Codex code review (B3):** Identifies a concrete structural cause: calls 2 and 3 replace the original plan documents with only "Generate 5 MORE levers… [name blacklist]" while inheriting prior assistant JSON. The model is therefore conditioned on its own previous output (including contaminated consequence text) with no grounding in the source plan.
- **Verdict:** Both claims are valid and complementary. Verifying in source: lines 196–209 confirm that `call_index > 1` replaces `prompt_content` with just the name-blacklist message, and lines 233–241 confirm that the full assistant response is appended to `chat_message_list` before the next call. This is a concrete structural cause, not just a fallback path. Codex's framing (B3) is correct and stronger than Claude's (S3).

### Disagreement 2: Ranking of run 52 (haiku) vs. run 49 (gpt-5-nano)

- **Insight Claude:** ranks run 52 #1 (full compliance, rich content, timeout resolved, strategic_rationale populated).
- **Insight Codex:** ranks run 49 #1 (best diversity, completeness, and trade-off explicitness).
- **Verdict:** Both are Tier A. The difference is verbosity trade-off. Run 52 averages ~550 chars/consequence and 1374 chars/summary but has one 6-lever response (B1 violation). Run 49 averages ~420 chars/consequence with zero structural violations. For production use, run 52's 1374-char summaries and 1060-char consequences (batch average) are overkill; run 49's 33K tokens/plan is an emerging timeout risk. Neither is clearly better — this disagreement reflects a genuine quality vs. cost trade-off, not a factual error.

### Disagreement 3: Race condition severity

- **Claude code review (B2):** Flags `set_usage_metrics_path` outside `_file_lock` as a confirmed bug causing usage-metric misrouting under `workers > 1`.
- **Codex code review:** Does not flag this race condition at all.
- **Verdict:** Claude is correct. Source lines 106 and 140 confirm `set_usage_metrics_path` calls are outside the lock while `dispatcher.add/remove` at lines 109 and 141–142 are inside it. The asymmetry is real. However the practical impact is limited to usage metrics (not output quality), so it is lower priority than the schema and chat-structure bugs.

## Top 5 Directions

### 1. Enforce the schema contract in code: constrain lever count and make `summary` required
- **Type**: code fix
- **Evidence**: Both code-review agents independently flag this (Claude B1/S1, Codex B1/B2). Confirmed in source: `levers: list[Lever] = Field(description="Propose exactly 5 levers.")` has no `min_length`/`max_length`; `summary: Optional[str] = Field(default=None, ...)`. Run 52 produces 16 levers (hong_kong_game). Null summaries: 14 (run 47), 10 (run 48), 4 (run 51).
- **Impact**: Eliminates silent over-count and null-summary artifacts for all models. Turns a prompt-compliance failure into a hard parse error that can be caught, logged, or retried. Affects every model that is tested.
- **Effort**: Low — two targeted changes in `identify_potential_levers.py`.
- **Risk**: Low. Making `summary` required could cause parse failures for models that currently produce null; this surfaces a real problem rather than hiding it. Adding `min_length=5, max_length=5` to `levers` could fail on a 6-lever response (run 52); the fallback is an error that can be retried or truncated.

### 2. Fix multi-call chat structure: re-include original plan documents in calls 2 and 3
- **Type**: code fix (prompt-construction logic)
- **Evidence**: Codex B3, confirmed in source lines 196–209 and 233–241. Calls 2 and 3 send only "Generate 5 MORE levers… [name blacklist]" without the original plan documents; the prior assistant JSON (including any contaminated field text) is accumulated into the conversation history. This is the most plausible structural cause of qwen3 contamination carry-over, verbosity drift in gpt-5-nano, and template reproduction across calls.
- **Impact**: Expected to reduce or eliminate contamination propagation between calls. Keeps all calls grounded in the source plan, reducing style-drift and self-reinforcing errors. Affects all multi-call models, particularly qwen3-30b (contamination) and gpt-5-nano (verbosity amplification).
- **Effort**: Medium — requires restructuring the `execute()` loop in `identify_potential_levers.py`. The fix is: on calls 2 and 3, instead of appending prior assistant JSON, start a fresh message list with system prompt + original user_prompt + explicit name-exclusion list.
- **Risk**: Medium. The current multi-turn approach successfully prevents exact name duplication across calls (0 cross-call duplicate names in all tested runs vs. 27 in baseline). Switching to fresh-context calls could weaken diversity unless the name-exclusion instruction is retained.

### 3. Add post-generation validator for consequence contamination
- **Type**: code fix (post-parse validation)
- **Evidence**: Both code-review agents recommend this (Claude I4, Codex I2). The field-description prohibition is confirmed in source (lines 38–39) but run 50 still shows 100% contamination. A model that ignores the description will not be fixed by rewording it; enforcement must happen after generation.
- **Impact**: Prevents contaminated levers from being written to `002-10-potential_levers.json` regardless of model. Could flag or reject all 75 contaminated levers in a run-50-style batch. Also documents a binary quality signal useful for model selection.
- **Effort**: Medium — add a check in `execute()` after `levers_cleaned` is assembled, before `save_clean()`. The detector pattern is simple: `"Controls" in text and "vs." in text` or `"Weakness:" in text` in `lever.consequences`.
- **Risk**: Low. False positives (valid consequences that happen to mention "Controls" in a different sense) are rare given the structured format. A warning-and-flag approach (rather than hard-reject) avoids disabling an entire run.

### 4. Fix race condition: move `set_usage_metrics_path` inside `_file_lock`
- **Type**: code fix (concurrency)
- **Evidence**: Claude code review B2, confirmed in source. `set_usage_metrics_path` at lines 106 and 140 is outside `_file_lock` while `dispatcher.add/remove` at lines 109 and 141–142 are inside it. Under `workers > 1`, thread A's metrics can be written to thread B's file or nulled while a call is in flight.
- **Impact**: Correct usage metrics for concurrent plan runs. The current bug silently misroutes or drops metrics, making per-model performance analysis unreliable when parallelism is used.
- **Effort**: Low — move two `set_usage_metrics_path` calls inside `_file_lock` blocks.
- **Risk**: Very low. The fix is mechanical; no logic changes.

### 5. Add minimum word count to options description + skip nemotron from runner
- **Type**: prompt change + code change
- **Evidence**: Both insight agents and both code-review agents flag llama3.1 options (Claude N3/I2, Codex I1). Nemotron failures flagged by both insight agents (Claude N1, Codex S5-implied, Claude code I7). Options current description: "a complete strategic approach (a full sentence with an action verb), not a label" — qualitative only. llama3.1 produces ~18-char labels.
- **Impact**: Adding "at least 15 words" to the options description gives weaker models a quantitative floor. However, llama3.1's token budget per call may be too tight for both full consequences and long options simultaneously — impact is uncertain. Skipping nemotron saves ~8 minutes per batch with zero upside loss (five consecutive 0/5 batches).
- **Effort**: Low for both changes.
- **Risk**: Low for nemotron skip (confirmed capability failure). For options word count: adds token pressure to already-strained models; llama3.1 might shorten consequences to compensate. Test on llama3.1 specifically before treating as universal.

## Recommendation

**Do Direction 1 first: enforce the schema contract.**

Specifically, make two changes in `identify_potential_levers.py`:

**Change A** — Constrain lever count to exactly 5 per response (line 74):
```python
# Before:
levers: list[Lever] = Field(
    description="Propose exactly 5 levers."
)

# After (Pydantic v2):
levers: list[Lever] = Field(
    min_length=5,
    max_length=5,
    description="Propose exactly 5 levers."
)
```

**Change B** — Make `summary` required (lines 77–80):
```python
# Before:
summary: Optional[str] = Field(
    default=None,
    description="Are these levers well picked? ..."
)

# After:
summary: str = Field(
    description="Are these levers well picked? Are they well balanced? Are they well thought out? Point out flaws. 100 words."
)
```

**Why first:**
- Highest confidence: two independent agents agree; source-confirmed; run outcomes are directly explained.
- Zero ambiguity: these are hard guarantees, not heuristics. A 6-lever response will fail at parse time; a null summary will fail at parse time. The failures are detectable, loggable, and retryable rather than silently corrupting output files.
- Lowest effort relative to impact: two targeted field changes, no logic restructuring.
- Broad coverage: affects every model tested (run 52 on lever count; runs 47/48/51 on summary), not just one pathological model.
- No side effects on the diversity mechanism (name-exclusion loop in `execute()` is unchanged).

Direction 2 (multi-call chat structure) is the second most important fix and addresses the root cause of qwen3 contamination propagation and gpt-5-nano verbosity drift more fundamentally. It should follow Direction 1 in the next iteration.

## Deferred Items

- **Direction 2 (multi-call chat structure):** High-impact but requires careful restructuring of `execute()`. Defer to next iteration; needs a dedicated A/B test to confirm that fresh-context calls maintain diversity while reducing contamination carry-over.

- **Direction 3 (post-generation validator):** Important defensive measure; implement after Direction 2 since restructuring the chat loop may reduce contamination enough to make this a redundancy rather than a necessity.

- **Direction 4 (race condition):** Low-effort correctness fix for `runner.py:106,140`. Can be done at any time alongside other work; does not affect output quality.

- **Direction 5, nemotron skip:** Add `--skip-model` or a `"skip": true` flag in model config. Trivial to implement; unblock it by adding the flag rather than removing the model config entirely, to preserve the ability to re-test if a new model version is released.

- **Direction 5, options word count:** Test in isolation on llama3.1 only first. If option quality improves without shortening consequences, apply generally. If llama3.1 compensates by shortening consequences (token budget transfer), this may not be worth the risk for other models.

- **`strategic_rationale: Optional[str]`:** Currently still optional (lines 70–73). Run 52 is the only confirmed non-null instance. If `strategic_rationale` is considered useful downstream, make it required alongside `summary`. Deferred because the data on its downstream utility is thin.

- **Per-call token cap for gpt-5-nano:** 33K tokens/plan is a risk for harder plans. Consider passing `max_tokens=3000–4000` per call. Defer until a timeout is actually observed; premature truncation risks malformed JSON.

- **Evaluation split (baseline resemblance vs. prompt-contract compliance):** Codex insight C4 raises this as a meta-level concern. The baseline is contaminated; scoring against it can penalize cleaner outputs. Worth implementing a contract-compliance scorer as a second evaluation axis. Deferred — requires new measurement infrastructure.
